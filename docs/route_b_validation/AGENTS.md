# AGENTS.md — route_b_validation/

## Context

This directory contains the **Route B validation** experiment: an offline
investigation into whether EEF-based supervision signals (instead of OSC
delta actions) can be replayed open-loop using built-in robosuite controllers.
**Result: built-in controller routes fail** (0.4 cm → 37-74 cm
end-of-trajectory error). A later Panda + WholeBodyMinkIK follow-up shows that
expert absolute EEF targets can be replayed with a third-party controller, but
the learned EEF policy still fails rollout.

The full report is in [`report.md`](./report.md). The follow-up adapter report
is in [`adapter_report.md`](./adapter_report.md). **You are reading this
file** for reproduction instructions. The experiment is fully self-contained
here — dataset, scripts, raw data, figures, and a one-shot shell wrapper.

## How to reproduce

From this directory:

```bash
MUJOCO_GL=egl uv run python compare_all.py    # ~5 min, writes results.json
uv run python plot_results.py                # writes figures/*.png
```

Inputs are hardcoded inside `compare_all.py`:
- Dataset: `third_party/robomimic/datasets/can/yq/image_v15_delta_eef.hdf5`
- 4 plans × 5 demos, ~300 control steps each

To use more demos, edit `N_DEMOS` near the top of `compare_all.py`.

## What the experiment does

For each of 4 (signal, controller) pairs, the script replays 5 demos
open-loop and records per-step:
- `desired_dpos` (cm): what the action intended (in meters)
- `actual_dpos` (cm): what the EEF actually moved
- `err_to_target`: per-step distance from replayed EEF to data's `next_eef_pos[t]`
- `err_to_orig`: per-step distance from replayed EEF to data's `obs_eef_pos[t+1]`

The 4 plans:

| Plan | Signal | Controller |
|---|---|---|
| A | `action[7]` (OSC delta) | OSC delta |
| B-1 | `delta_eef_action[7]` (real EEF delta in OSC units) | OSC delta |
| B-2 | `next_eef_pos[3]` (absolute target) | OSC absolute |
| C | `next_eef_pos[t] - eef[t]` (cumulative delta in meters) | IK delta |

Only built-in Plan A passes (0.4 cm end error). See `report.md` for the data
tables, plots, and structural analysis of why each built-in alternative fails.
For the later third-party Mink controller follow-up, see
`outputs/route_b_validation/panda_mink_controller/STAGE_CONCLUSION.md`.

## Common questions

**Q: Why is the dataset 5 demos, not all 200?**
A: 5 is enough to demonstrate the failure pattern. The same 5 demos are used
for all 4 plans so the comparison is apples-to-apples. To use more, edit
`N_DEMOS` near the top of `compare_all.py`.

**Q: Can I run just one plan?**
A: Yes — `compare_all.py` has per-plan sections. Comment out the others,
or import it and call the inner functions directly.

**Q: The report cites a metric I don't see in the JSON.**
A: All metrics are documented in `README.md` under "What the data files contain".

**Q: I want to re-derive the figures without rerunning the experiment.**
A: `uv run python plot_results.py` reads `results.json` and writes plots
into `figures/`. No sim involved.

## Maintenance

- **Adding a plan**: edit `compare_all.py`, add a section in the per-plan
  format. Update the `plan_order` list in `plot_results.py` and the
  `PLAN_LABELS` / `PLAN_COLORS` dicts. Add a row to the result table in
  `report.md`.
- **Adding a metric**: record it in `replay()` in `compare_all.py`, add to
  the `replay_traj` save block, update `README.md` schema docs, add to
  `report.md` tables.
- **Dataset path changes**: edit line 22 of `compare_all.py`. All paths in
  this directory are derived from `__file__`, so the directory remains
  portable as long as the repo layout is preserved.
